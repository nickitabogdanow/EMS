# EMS — веб-приложение для CSV (freq / ampl)

Репозиторий: [github.com/nickitabogdanow/EMS](https://github.com/nickitabogdanow/EMS)

```bash
git clone https://github.com/nickitabogdanow/EMS.git
cd EMS
```

Веб-интерфейс на **FastAPI** для загрузки спектров в формате CSV (колонки `freq`, `ampl`): **вычитание** двух кривых, **объединение** по частоте, интерактивные **графики Plotly**, отдельная выгрузка результата и HTML.

## Возможности

- Загрузка **нескольких CSV** в браузере (drag-and-drop или выбор файлов), выбор пары файлов для операции.
- **Вычитание:** A − B или B − A по совпадающим значениям `freq`.
- **Объединение:** объединение частот из обоих файлов с сортировкой; при совпадении `freq` — среднее, значение из A или из B.
- **График:** отображение исходных A/B и результата (чекбоксы); по умолчанию можно оставить только одну кривую.
- **Подсветка при вычитании:** полосы на графике, где |разность| не меньше порога; **зелёный** — разность &gt; 0, **красный** — &lt; 0.
- Скачивание **result.csv** / **merged.csv** отдельным запросом после расчёта и **chart.html** (автономная страница с графиком).
- Темы оформления: тёмная, светлая, Barbie, «Единороги» (фон SVG).
- Запуск через **Docker** или локальный **Python**.

## Требования

- Python **3.12+** (рекомендуется) или образ Docker с Python 3.12.

## Быстрый старт (локально)

```bash
cd EMS   # или каталог, куда вы склонировали проект
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

Запуск (из корня проекта):

```bash
./ems          # macOS / Linux
python run.py  # или явно
```

`run.py` и скрипты `ems` / `ems.bat` — это **dev-runner**: они поднимают приложение с `reload=True` и при занятом `PORT` ищут следующий свободный порт.

Либо:

```bash
uvicorn app.main:app --host 127.0.0.1 --port 8000
```

Этот вариант более предсказуем для CI, Docker и ручных smoke-test сценариев.

Откройте в браузере: **http://127.0.0.1:8000/**

> Страницу нужно открывать **через HTTP**, а не как файл `index.html` с диска — иначе запросы к API не работают.

## Production baseline

Текущая production-модель для `EMS` — **один инстанс / один worker** внутри доверенной сети. Это осознанное ограничение:

- результаты скачивания хранятся во **временном in-memory store**;
- `Dockerfile` запускает `uvicorn` с `--workers 1`;
- старые результаты могут быть вытеснены из памяти раньше TTL, если store достигнет лимитов.

Рекомендуемый запуск без dev-runner:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1
```

Служебные endpoint'ы:

- `GET /healthz` — быстрый liveness check;
- `GET /ready` — готовность приложения и краткая статистика `result_store`.

Если сервис публикуется за reverse proxy, задайте таймауты и лимит тела запроса на стороне прокси и **не увеличивайте число worker'ов**, пока не будет заменён `in-memory` store.

## Формат CSV

- Разделитель: запятая или точка с запятой.
- Заголовок с колонками **`freq`** и **`ampl`** (допускаются варианты `frequency`, `amplitude`, `f`, `a` и т.п.).
- Кодировка: **UTF-8**.

## Большие файлы (сотни тысяч точек)

Ответ API с полным графиком в JSON сильно разрастается и тормозит браузер. Для **отрисовки** сервер автоматически **прореживает** ряды (min/max по частотным бакетам, форма спектра сохраняется). Полный CSV результата **не встраивается в JSON** и скачивается отдельным запросом, поэтому обычный ответ для UI остаётся компактным.

Лимит точек на кривую на графике: переменная окружения **`EMS_MAX_PLOT_POINTS`** (по умолчанию **14000**, допустимый диапазон 2000…500000).

Если пользователь запрашивает режим **без прореживания**, сервер разрешит его только до лимита **`EMS_MAX_FULL_PLOT_POINTS`** (по умолчанию **500000** точек на ряд). Более тяжёлые запросы отклоняются с `400`, чтобы браузер не зависал.

Подсветка больших разностей строится по **тому же прореженному** ряду, что и линия результата; близкие полосы одного знака **склеиваются**. Максимум фигур в Plotly: **`EMS_MAX_HIGHLIGHT_SHAPES`** (по умолчанию **700**), чтобы браузер не зависал.

Готовый результат для скачивания хранится во временном in-memory store с TTL. Время жизни: **`EMS_RESULT_TTL_SECONDS`** (по умолчанию **3600** секунд, допустимый диапазон 60…86400). Если скачать CSV слишком поздно, API вернёт `410 Gone`, и расчёт нужно повторить.

Дополнительные safety-лимиты:

| Переменная | По умолчанию | Назначение |
|-----------|--------------|------------|
| `EMS_MAX_CSV_ROWS` | `400000` | максимум числовых строк в одном CSV |
| `EMS_MAX_FULL_PLOT_POINTS` | `500000` | максимум точек на ряд для режима без прореживания |
| `EMS_MAX_UPLOAD_BYTES` | `20971520` | максимум байт на один загружаемый файл |
| `EMS_MAX_REQUEST_BODY_BYTES` | `47185920` | максимум `Content-Length` всего POST-запроса |
| `EMS_MAX_RESULT_POINTS` | `500000` | максимум точек в одном временно сохранённом результате |
| `EMS_MAX_STORED_RESULTS` | `32` | максимум результатов в in-memory store |
| `EMS_MAX_STORED_POINTS_TOTAL` | `2000000` | суммарный лимит точек во всём in-memory store |
| `EMS_LOG_LEVEL` | `INFO` | уровень логирования (`DEBUG`, `INFO`, `WARNING`, `ERROR`) |
| `EMS_TRUSTED_HOSTS` | пусто | comma-separated список host header'ов для `TrustedHostMiddleware` |

Пример:

```bash
EMS_MAX_PLOT_POINTS=20000 EMS_MAX_CSV_ROWS=500000 python run.py
```

В Docker добавьте в `docker-compose.yml` под `environment:`.

## Ограничения и допущения

- CSV разбираются **потоково из upload-потока** без полного `read()` в строку, но итоговые ряды и результат всё равно живут **в памяти процесса**. Это удобно для локального инструмента, но не рассчитано на тяжёлую многопользовательскую нагрузку.
- Сопоставление частот идёт по **точному значению `float`**. Если в двух CSV частоты отличаются на тысячные из-за форматирования или округления, они будут считаться разными.
- Если одна и та же `freq` повторяется **внутри одного файла**, парсер сохраняет последнее встретившееся значение амплитуды для этой частоты.
- Для графика данные могут быть прорежены, но скачиваемый CSV остаётся полным.
- Результат скачивания не сохраняется на диск сервера по умолчанию: он временно лежит в памяти и доступен по `result_id`, пока не истечёт TTL или пока store не вытеснит старые результаты по лимиту памяти.
- Сервис рассчитан на **один процесс приложения**. Запуск нескольких worker'ов или нескольких инстансов без внешнего shared-store приведёт к непредсказуемым ошибкам скачивания.
- `GET /api/download/{result_id}` безопасен только в рамках доверенной сети: `result_id` не привязан к отдельной пользовательской сессии.

python -m unittest discover -s tests -v

- `csv_utils` — парсинг, вычитание и merge-политики;

## API (кратко)

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/` | Главная страница |
| GET | `/healthz` | Быстрый liveness check |
| GET | `/ready` | Готовность приложения и статистика `result_store` |
| POST | `/api/analyze` | Вычитание: `file_a`, `file_b`, `operation` (`a_minus_b` / `b_minus_a`), флаги отображения, `highlight_threshold` (дБ, 0 = без подсветки). В ответе приходят `figure`, `result_id`, `result_filename`, `result_download_url` |
| POST | `/api/merge` | Объединение: `duplicate_policy` (`average` / `a` / `b`), файлы и флаги графика. В ответе приходят `figure`, `result_id`, `result_filename`, `result_download_url` |
| GET | `/api/download/{result_id}` | Скачать полный CSV результата по `result_id`; если результат уже истёк, возвращается `410` |

Файлы в запросе **не сохраняются на диск** сервера. CSV разбираются из upload-потока, а готовый результат временно хранится в памяти процесса и скачивается отдельным запросом.

Ошибки API возвращаются в стабильном формате:

```json
{
  "detail": "Понятное сообщение об ошибке",
  "request_id": "4f8d...",
  "error": {
    "code": "invalid_parameter",
    "message": "Понятное сообщение об ошибке",
    "request_id": "4f8d..."
  }
}
```

`X-Request-ID` также добавляется в каждый ответ и в JSON-логи, чтобы проще сопоставлять ошибки с серверными логами.

## Короткий runbook

- `GET /healthz` отвечает `200` — процесс жив.
- `GET /ready` показывает состояние приложения и текущую загрузку `result_store`.
- Ошибка `410 Gone` на `/api/download/{result_id}` означает, что результат истёк по TTL или был вытеснен по лимиту памяти; расчёт нужно повторить.
- При росте памяти сначала уменьшайте `EMS_MAX_CSV_ROWS`, `EMS_MAX_RESULT_POINTS` и `EMS_MAX_STORED_POINTS_TOTAL`, а уже потом увеличивайте Docker memory limit.
- Если нужен внешний доступ, ставьте reverse proxy с auth, body limits и таймаутами перед приложением.
